## 中ボタンドラッグ不具合の原因と修正（Phase 1）

旧実装の問題:
1. 中クリックがブラウザ風「オートスクロール」実装で、押しながら動かすパンではなかった
2. 仮想キャンバスが 2600×1800×Zoom 固定で、ウィンドウが大きい/ズーム1.0以下だと
   スクロール可能量が0になり**一切パンできなくなる**（「画像設置後動かない」の正体）

修正: 中ボタン＝ドラッグパンに変更し、固定仮想キャンバスを廃止（無限キャンバス化）。
スクロール範囲は「全画像の外接矩形∪ビューポート」から動的算出。コンテンツを完全に
見失わない緩いクランプのみ実施。`timeBeginPeriod(1)` 常用も廃止。
→ tools\pan_test.ps1 でE2E検証済み。

## Phase 5: 性能計測結果と言語判断

計測環境: Windows 11 / 20コアCPU / Debugビルド / 4K(3840×2160)PNG×2 + 2.5K PNG + WebP実ファイル

| 指標 | 実測値 | 評価 |
|---|---|---|
| 起動→ウィンドウ表示 | 927ms | ○ |
| メモリ (4K級4枚展開後) | 213MB | ○ 内約100MBは画像ピクセル自体（言語非依存） |
| **アイドルCPU** | **0.0%** | ◎ ゲーム併用時の影響なし |
| パン連打中CPU (125msg/s) | 1コア相当92% (全体4.6%) | △ GDI+の上限。操作中のみで通常60Hz入力なら約半分 |
| 配布サイズ (自己完結単一exe) | 62.8MB | ○ |
| 配布サイズ (ランタイム依存) | 0.3MB | ◎ 要 .NET 8 Desktop Runtime |

### 判断: C# (WinForms) 継続。Rust/C++リライトは実施しない

理由:
1. **要件「ゲーム併用時に体感影響なし」を満たした。** オーバーレイは変更時のみ再描画で、
   ゲームプレイ中（配置操作をしていない間）のCPUは0%。
2. メモリの大半はデコード済みビットマップで、これはRust/C++でも同量。言語変更で減るのは
   ランタイム分(約80MB)のみ。
3. リライトで得られる主な利得はGPU描画によるドラッグ中の滑らかさだが、ドラッグ中は
   既に低品質補間へ切替済みで実用上問題ない。エクスプローラーツリー等の資産を捨てて
   数週間のリライトを行う対価に見合わない。

### 将来リライトを再検討するトリガー
- 高ズーム・大量画像でのドラッグのカクつきが実使用で問題になった場合
  → まずC#のまま描画層のみ Direct2D (Vortice.Windows) 化を検討（UIは無傷で済む）
- それでも不足する場合のみ Rust (windows-rs + Direct2D/WIC) での全面リライト
  （その際は方針どおり F:\dev に新規フォルダを作成）

## 残タスク・既知の制限

- フルスクリーン排他モードのゲーム上には表示不可（OS仕様。ボーダレス必須 — README明記済み）
- 削除した画像のビットマップはドキュメントを閉じるまで保持（Undo復元のため。意図的仕様）
- サムネイル一覧は1フォルダ最大300ファイルまで表示
- クリップスタジオ(.clip)は内蔵サムネイル抽出のみ現実的（未実装）
- テーマ切替時、Windows標準コンボボックス等一部コントロールの色が完全には追従しない
